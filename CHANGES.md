# Changelog

## v1.0 тАФ 2026-05-01  Initial release

### Original specification

> Use the database of allmydox.
> Take a database that is generated by allmydox and create a GUI with the
> following features:
> 0. Implemented in Python
> 1. The user can input a word
> 2. The word and the nouns, names and verbs occurring in the same sentence
>    should be visualised in 3 word clouds (see 3.)
> 3. The words are represented by three word clouds: 1. Names  2. Nouns  3. Verbs
> 4. The metric between the words is the frequency of each word and the frequency
>    of being together in a sentence or a paragraph.
>    Sentence counts 30% higher than paragraph.
> 5. If possible show the 3 word clouds even if the user did not give an input
> 6. If the user left-clicks on a word in the cloud the cloud should be centred
>    around this word.
> 7. If the user right-clicks on a word, show a list of the documents where it
>    occurs on the right of the GUI window.
> 8. When the user clicks on a document in the document list it should be opened
>    and scrolled to the first occurrence of the word.
>
> The software shall run under Linux and Windows.

---

### Implementation decisions

**GUI framework тАФ PyQt6**  
Cross-platform, actively maintained, native look on both Linux and Windows.

**Word cloud rendering тАФ `wordcloud` + matplotlib**  
The `wordcloud` library computes an optimal layout (word positions, sizes,
orientations). Instead of exporting a flat image, each word is placed as an
individual matplotlib text object so that mouse-click hit-testing can
identify which word was clicked. This enables left- and right-click
interaction without any external JavaScript or browser widget.

**Scoring formula**  
`score = sentence_co_occurrences ├Ч 1.3 + paragraph_co_occurrences ├Ч 1.0`  
Matches the specification: sentence co-occurrence is weighted 30 % higher.

**Global view on startup (spec point 5)**  
When no search term has been entered, the three clouds show the top 150
words by total occurrence frequency across all documents.

**Left-click (spec point 6)**  
Sets the clicked word as the new search term and immediately re-queries
the database, recentring all three clouds around the new word.

**Right-click (spec point 7)**  
Populates a document panel on the right side of the window with one entry
per (document, page) where the word occurs. The panel is hidden until
first used.

**Document viewer (spec point 8)**  
A built-in viewer is used instead of the system's default application,
because there is no single cross-platform command to open a file *at a
specific page*:
- **PDF** тАФ pages are rendered as images via `pymupdf`. The first page
  containing the word is shown on open; all occurrences of the word on the
  visible page are highlighted in yellow. Previous / Next page buttons allow
  manual navigation.
- **DOCX / TXT** тАФ content is shown as plain text in a scrollable editor;
  the cursor is placed at the first occurrence of the word.

**Font handling**  
`matplotlib.font_manager` is used to locate a suitable system font at
runtime (DejaVu Sans тЖТ Arial тЖТ Liberation Sans тЖТ fallback). No font is
bundled; the selection works on both Linux and Windows.

**Performance тАФ database indexes**  
The `noun_sentence` and `noun_paragraph` tables produced by allmydox can
contain tens of millions of rows. Without dedicated indexes, co-occurrence
queries would require full table scans and take 30+ seconds. On first
launch, `query.ensure_indexes()` is called in a background thread and
creates 9 indexes on the co-occurrence and vocabulary tables:

| Index | Table | Columns |
|---|---|---|
| idx_noun_occ_noun | noun_occurrences | nounID |
| idx_name_occ_name | name_occurrences | nameID |
| idx_verb_occ_verb | verb_occurrences | verbID |
| idx_noun_sent_occ1 | noun_sentence | occ1_type, occ1_id |
| idx_noun_sent_occ2 | noun_sentence | occ2_type, occ2_id |
| idx_noun_para_occ1 | noun_paragraph | occ1_type, occ1_id |
| idx_noun_para_occ2 | noun_paragraph | occ2_type, occ2_id |
| idx_nvs_noun_occ | noun_verb_sentence | noun_occ_type, noun_occ_id |
| idx_nvs_verb_occ | noun_verb_sentence | verb_occ_id |

All use `CREATE INDEX IF NOT EXISTS`, so subsequent launches skip the step
instantly. The same indexes were also added to `allmydox/db.py` so that
newly created databases include them from the start.

**Query architecture**  
All co-occurrence lookups use SQL JOIN queries rather than Python loops
with individual per-row lookups. This keeps the full aggregation work
inside SQLite and avoids N+1 query patterns that would be prohibitively
slow on large databases.

## v1.1 тАФ 2026-05-01  Pre-computed score cache

### Problem

Co-occurrence queries against the raw `allmydox.db` took 90тАУ120 seconds per
search on a 5 GB database (48 million rows in `noun_sentence` / `noun_paragraph`).
Even with 9 dedicated indexes, SQLite must JOIN, group, and aggregate millions of
rows on every keystroke тАФ too slow for interactive use.

### Solution тАФ `cache.py`

A new `cache.py` module pre-computes **all** co-occurrence scores into a separate
flat SQLite file (`<dbname>.cache.db`), built once on first launch.

**Build process**

23 SQL `INSERT тАж SELECT` steps run via `ATTACH DATABASE` so no data is copied
out of SQLite:

| Steps | Description |
|---|---|
| 8 ├Ч noun_sentence | nounтЖФnoun, nounтЖФname, nameтЖФnoun, nameтЖФname тАФ forward and reverse, weight ├Ч1.3 |
| 8 ├Ч noun_paragraph | same four combinations тАФ forward and reverse, weight ├Ч1.0 |
| 4 ├Ч noun_verb_sentence | nounтЖФverb, nameтЖФverb тАФ both directions, weight ├Ч1.3 |
| 3 ├Ч word_freq | noun, name, verb occurrence counts for the global cloud view |

A final consolidation step creates the permanent `cooccurrence` table from the
raw inserts using `SELECT тАж SUM(score) GROUP BY`, then adds an index on
`lower(src_word)` and drops the staging table.

**Results on a 5 GB / 6,097-document database**

| Metric | Before | After |
|---|---|---|
| Search response time | 90тАУ120 s | 2тАУ5 ms |
| Cache build time (once) | тАФ | ~4.5 min |
| Cache file size | тАФ | 527 MB |

**App integration**

`app.py` was updated with a two-stage startup:

1. `_IndexWorker` ensures the 9 raw-DB indexes exist (instant on subsequent launches).
2. `_CacheWorker` builds the cache if not yet present, showing a `QProgressDialog`
   with per-step progress. Subsequent launches skip directly to the cloud view.

All search and global-frequency lookups now go through `cache_mod.cooccurrences()`
and `cache_mod.global_frequencies()`. Document-occurrence lookups (right-click
panel) still use the raw database тАФ those queries are already fast because they
hit a small result set by document ID.

## v1.2 тАФ 2026-05-06  Interactive database and documents folder selection

### Problem

The database path had to be supplied as a command-line argument, and the
folder paths stored inside the database were absolute paths recorded at
index time by allmydox. Both requirements made the application awkward to
use when the database was handed to a new user or the document files had
been moved to a different machine or directory.

### Changes

**Interactive database picker**

`main.py` no longer exits with an error when no database path is given.
Instead it tries the following in order:

1. The path passed as a positional argument (unchanged behaviour).
2. `allmydox.db` in the current working directory.
3. A `QFileDialog` file picker so the user can browse to the database.

**Documents folder override**

A new `--docs FOLDER` command-line argument and a **File > Set Documents
FolderтАж** menu item let the user specify where document files are located.
When a document is opened, the application first tries the path stored in
the database; if that file does not exist it looks for the filename inside
the documents folder.

**File menu**

A **File** menu was added to the main window:

| Item | Shortcut | Description |
|---|---|---|
| Open DatabaseтАж | Ctrl+O | Browse to a different database; opens a new window |
| Set Documents FolderтАж | тАФ | Set the fallback folder for document files |

**Dark-theme styling** was extended to cover the new menu bar and menus.

### Files changed

| File | Change |
|---|---|
| `main.py` | Interactive file picker on startup; new `--docs` argument |
| `app.py` | `docs_folder` parameter; `_build_menu()`; `_on_open_database()`; `_on_set_docs_folder()`; fallback path logic in `_on_doc_clicked()` |

## v1.3 тАФ 2026-05-06  Incremental cache update

### Problem

Every time new documents were added to the source database and the cache
file was deleted to pick them up, the full 4.5-minute build ran again from
scratch тАФ even though the vast majority of co-occurrence data was unchanged.

### Solution

The cache now stores a `max_file_id` watermark in a `meta` table written at
the end of each build or update. On startup, `needs_update()` compares this
value against `MAX(fileID)` in the source database. If new documents exist,
only their co-occurrence rows are processed and merged into the existing
cache tables instead of rebuilding from scratch.

**Incremental build steps**

`_build_steps()` accepts a `min_file_id` parameter. When set, each SQL step
adds a `WHERE` clause that restricts the occurrence joins to rows with
`fileID > min_file_id`, capturing:

- New-document ├Ч new-document co-occurrences
- New-document ├Ч existing-document co-occurrences

The delta rows are inserted into staging tables (`raw_delta`, `freq_delta`),
consolidated with `GROUP BY`, then merged into the live `cooccurrence` and
`word_freq` tables.

**`word_freq` upsert**

`word_freq` has a `UNIQUE(word, kind)` constraint. The merge step uses
`INSERT тАж ON CONFLICT DO UPDATE SET freq = freq + excluded.freq` so word
frequencies are accumulated in place with no duplicate rows. This keeps
`global_frequencies()` as a plain index scan with no aggregation overhead.

**Startup flow**

```
_IndexWorker (indexes exist?)
    тФФтФА cache exists?
         тФЬтФА needs_update()? тЖТ _CacheWorker(mode="update") тЖТ show clouds
         тФЬтФА no              тЖТ show clouds immediately
         тФФтФА cache missing   тЖТ _CacheWorker(mode="build")  тЖТ show clouds
```

**Note:** existing `.cache.db` files built before v1.3 have no `meta` table
and no `UNIQUE` constraint on `word_freq`. They continue to work as before
but will not receive incremental updates. Delete the `.cache.db` file once
to trigger a fresh build that enables the new behaviour.

### Files changed

| File | Change |
|---|---|
| `cache.py` | `needs_update()`; `update()`; `_build_steps()` gains `min_file_id`, `raw_table`, `freq_table`; `UNIQUE(word, kind)` on `word_freq`; `meta` table in `build()` |
| `app.py` | `_CacheWorker` gains `mode` parameter; `_start_cache_build()` replaced by `_start_cache_worker(mode)`; `_on_indexes_ready()` calls `needs_update()` |

## v1.4 тАФ 2026-05-07  Configurable paths; manual cache rebuild

### Problem

The database path, documents folder, and cache file location were either
hard-coded or had to be re-supplied on every launch. The cache also
auto-updated silently on startup whenever new documents were found, giving
the user no control over when a potentially long rebuild happens.

### Changes

**Persistent settings тАФ `config.py`**

A new `config.py` module stores the three user-configurable paths in
`~/.config/findethedox/config.json`. Settings are written whenever the user
changes a path and are read on the next launch as defaults.

**Settings menu**

A new **Settings** menu replaces the old **File > Set Documents FolderтАж**
item and exposes both path options:

| Item | Description |
|---|---|
| Set Documents FolderтАж | Fallback folder for document files |
| Set Cache FileтАж | Location of the pre-computed cache database |

**Rebuild Cache toolbar button**

A toolbar button labelled **Rebuild Cache** triggers a full cache rebuild
after a confirmation dialog. This replaces the previous auto-update
behaviour: the application no longer silently rebuilds or updates the cache
on startup.

**Startup flow change**

`_on_indexes_ready()` no longer calls `needs_update()`. If a cache file is
present it is used immediately; if no cache exists the initial build runs
as before.

```
_IndexWorker (indexes exist?)
    тФФтФА cache exists? тЖТ connect and show clouds
    тФФтФА cache missing тЖТ _CacheWorker(mode="build") тЖТ show clouds

"Rebuild Cache" button тЖТ confirmation тЖТ _CacheWorker(mode="build")
```

**CLI**

`main.py` gains a `--cache FILE` argument to override the cache path from
the command line. On startup the priority order is:
CLI arg > saved config > auto-derived path (next to the database file).

### Files changed

| File | Change |
|---|---|
| `config.py` | New module: `load()` / `save()` for `~/.config/findethedox/config.json` |
| `main.py` | Loads config on startup; new `--cache` argument; passes `docs_folder` and `cache_path` to `MainWindow` |
| `app.py` | `MainWindow` accepts optional `cache_path`; Settings menu; Rebuild Cache toolbar button; `_save_config()`; startup no longer auto-updates cache |

## v1.5 тАФ 2026-05-07  Non-blocking search

### Problem

Every search тАФ triggered by pressing Enter, left-clicking a cloud word, or
right-clicking for the document list тАФ ran synchronously on the Qt main
thread. The co-occurrence lookup hits the cache (fast, ~5 ms), but the
document-occurrence query issues three separate JOIN queries against the raw
5 GB database and can take several seconds. During that time Qt's event loop
was blocked, causing the OS to display a "not responding" dialog.

### Solution тАФ `_SearchWorker`

A new `_SearchWorker(QThread)` class runs both queries in a background
thread and delivers results back to the main thread via signals:

| Signal | Payload | Triggered when |
|---|---|---|
| `cooc_ready` | list of result dicts, word | co-occurrence rows found in cache |
| `docs_ready` | list of `DocOccurrence`, word | document list ready from raw DB |
| `not_found` | word | cache returned no results |
| `error` | message | exception in worker thread |

The worker opens its own database connections (SQLite connections must not
be shared across threads), runs the queries, converts `sqlite3.Row` objects
to plain dicts before crossing the thread boundary, then emits the signals.

The synchronous `_load_cooccurrences()` and `_show_documents()` methods were
replaced by `_start_search(word, docs_only=False)`, which creates and starts
a `_SearchWorker`. Result handlers `_on_cooc_ready()` and `_on_docs_ready()`
update the clouds and document panel on the main thread.

`_load_global()` (startup / empty search box) continues to run on the main
thread because it queries the already-open in-memory cache connection and
completes in < 5 ms.

### Files changed

| File | Change |
|---|---|
| `app.py` | Added `_SearchWorker`; replaced synchronous search methods with `_start_search()`, `_on_cooc_ready()`, `_on_docs_ready()`; removed `WaitCursor` blocks from search path |
